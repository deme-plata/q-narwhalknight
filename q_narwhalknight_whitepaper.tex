\begin{abstract}
Quillon is deployable private settlement infrastructure---a post-quantum secure, Tor-native consensus stack designed for institutions, operators, and sovereign networks that require verifiable asset transfer without dependence on third-party chains or custodians. Built on DAG-Knight consensus with Narwhal mempool integration, Quillon achieves sub-50ms finality at over 48,000 transactions per second while maintaining mandatory network-layer privacy through dedicated Tor circuits. The system implements a phased cryptographic agility model transitioning from classical Ed25519 signatures to lattice-based Dilithium5 post-quantum cryptography, ensuring long-term settlement integrity against both classical and quantum adversaries. This paper presents the architecture, security model, and deployment characteristics of Quillon as production-grade settlement rails.
\end{abstract}

\section{Introduction}

\subsection{The Settlement Problem}

Digital asset settlement today depends on public, permissionless chains that were designed for transparency, not privacy. Institutions that require confidential settlement---central banks exploring CBDC rails, commodity exchanges settling physical delivery contracts, sovereign wealth funds managing reserve assets---face a structural gap: existing blockchain infrastructure either exposes transaction graphs to global surveillance, or relies on trusted intermediaries that reintroduce the counterparty risk distributed ledgers were meant to eliminate.

Meanwhile, the approaching reality of cryptographically relevant quantum computers threatens the signature schemes that secure every major blockchain in production. No deployed settlement system offers a credible migration path from classical to post-quantum cryptography without requiring a full network reset.

Quillon addresses both problems simultaneously. It is private settlement infrastructure: a deployable, sovereign-grade consensus stack with three defining properties:

\begin{enumerate}
    \item \textbf{Privacy by architecture}: All peer-to-peer communication routes through dedicated Tor circuits. Transaction mixing, Dandelion++ propagation, and zero-knowledge proofs are not optional add-ons---they are mandatory protocol layers. The network does not leak metadata.
    \item \textbf{Post-quantum security}: Cryptographic agility is built into the consensus layer. The system transitions from Ed25519 to Dilithium5 lattice-based signatures via height-gated upgrades, preserving historical block validity while hardening future settlement against quantum attack.
    \item \textbf{Deployable sovereignty}: Quillon is not a token or a hosted service. It is infrastructure that operators deploy, configure, and control. A central bank, a commodity exchange, or a private institution runs its own Quillon network with its own genesis, its own validators, and its own governance---without permission from any external chain.
\end{enumerate}

\subsection{What Quillon Is}

Quillon is a complete settlement stack. It includes:

\begin{itemize}
    \item \textbf{DAG-Knight Consensus}: A directed acyclic graph consensus mechanism achieving zero-message-complexity Byzantine fault tolerance, with VDF-based leader election and adaptive confirmation depth
    \item \textbf{Narwhal Mempool}: High-throughput transaction dissemination with reliable broadcast (Bracha's protocol), achieving ordered, deduplicated transaction delivery at scale
    \item \textbf{Tor-Native Networking}: Mandatory onion-routed peer communication with 4 dedicated circuits per validator, rotated each epoch, seeded by quantum-grade entropy

    \item \textbf{Integrated Settlement Services}: Native DEX with AMM liquidity pools, oracle network for external price feeds, and programmable settlement logic via WASM smart contracts
    \item \textbf{Operational Tooling}: Zero-downtime rolling deployment, pre-flight database verification, automatic fork detection, and real-time consensus monitoring

\section{Consensus Mechanism}

\subsection{DAG-Knight Algorithm}

The DAG-Knight consensus algorithm operates on a directed acyclic graph where each vertex represents a block of transactions. The algorithm achieves consensus through the following steps:

\begin{algorithm}
\caption{DAG-Knight Consensus}
\begin{algorithmic}[1]
\STATE \textbf{Input:} Vertex $v$ with parents $P(v)$
\STATE \textbf{Output:} Consensus decision
\STATE Broadcast vertex $v$ to all validators
\STATE Collect acknowledgments from $f+1$ validators
\STATE Compute anchor election using VDF
\IF{$v$ is elected anchor}
    \STATE Finalize all ancestors of $v$
\ENDIF
\STATE Continue to next round
\end{algorithmic}
\end{algorithm}

\section{Mining Architecture}

Quillon employs a hybrid BLAKE3+VDF proof-of-work scheme that is both GPU-friendly and ASIC-resistant. The mining function chains 100 sequential BLAKE3 hashes per nonce candidate, combining memory-hard initial hashing with a verifiable delay function (VDF) tail that prevents unbounded parallelism.

\subsection{Hash Function: BLAKE3 + VDF Chain}

Each mining attempt evaluates:

\begin{algorithm}
\caption{BLAKE3+VDF Mining (per nonce candidate)}
\begin{algorithmic}[1]
\REQUIRE Challenge hash $C \in \{0,1\}^{256}$, nonce $n \in \mathbb{Z}_{2^{64}}$, target $T$
\STATE $\mathit{input} \leftarrow C \| n_{\mathrm{LE}}$ \COMMENT{40 bytes: 32-byte challenge + 8-byte little-endian nonce}
\STATE $h \leftarrow \mathrm{BLAKE3}(\mathit{input})$ \COMMENT{Initial hash (40-byte single-block)}
\FOR{$i = 1$ \TO $99$}
    \STATE $h \leftarrow \mathrm{BLAKE3}(h)$ \COMMENT{VDF chain: 32-byte single-block hash}
\ENDFOR
\IF{$h < T$}
    \RETURN $(n, h)$ \COMMENT{Valid solution}
\ENDIF
\end{algorithmic}
\end{algorithm}

The 100-round VDF chain ensures that each nonce requires sequential computation, bounding the advantage of massively parallel hardware. A single BLAKE3 evaluation is fast ($\sim$3 ns on modern CPUs), but 100 sequential rounds per candidate create a minimum latency floor that limits ASIC speedup to constant factors over GPUs.

\subsection{GPU Mining Pipeline}

The standalone miner (\texttt{q-miner}) implements a high-performance OpenCL backend for GPU-accelerated mining. The GPU pipeline operates as follows:

\begin{enumerate}
    \item \textbf{Kernel compilation}: The BLAKE3+VDF kernel is compiled from OpenCL~C source at first launch. Compiled binaries are cached to \texttt{\textasciitilde/.config/q-miner/kernel-cache/\{hash\}.clbin} for subsequent startups (cache key = SHA-256 of source + device name + driver version), reducing startup from 2--10\,s to $<$100\,ms.

    \item \textbf{Persistent buffer allocation}: Five GPU buffers are allocated once per device and reused across all dispatches, eliminating per-dispatch IPC round-trips:
    \begin{itemize}
        \item \texttt{challenge\_buf}: $8 \times \mathtt{u32}$ (\texttt{\_\_constant} address space, hardware constant cache)
        \item \texttt{target\_buf}: 32 bytes (\texttt{\_\_global}, read-only)
        \item \texttt{found\_nonce\_buf}, \texttt{found\_hash\_buf}, \texttt{found\_flag\_buf}: write-only result buffers
    \end{itemize}

    \item \textbf{Conditional upload}: Challenge and target data are re-uploaded only when the block changes. Within a block, dispatch overhead is zero-copy.

    \item \textbf{Non-blocking dispatch}: Buffer writes use \texttt{CL\_FALSE} (asynchronous) on an in-order command queue. The GPU sequences writes before kernel execution; the CPU proceeds immediately without stalling.

    \item \textbf{Adaptive work sizing}: Each GPU independently auto-tunes its global work size within $[2^{16}, 2^{23}]$ items to maintain dispatch latency in the $[100, 400]$\,ms window. Startup calibration benchmarks four work sizes (64K, 256K, 1M, 4M) to find the optimal starting point.
\end{enumerate}

\subsection{Kernel Optimizations}

The OpenCL kernel incorporates several micro-optimizations:

\begin{itemize}
    \item \textbf{CPU-side word conversion}: The 32-byte challenge is converted to $8 \times \mathtt{u32}$ little-endian words on the CPU once, eliminating per-work-item byte-to-word unpacking on the GPU.
    \item \textbf{Constant cache}: The challenge buffer uses \texttt{\_\_constant} address space, enabling hardware broadcast from the GPU's dedicated constant cache (one global memory read per work-group instead of per work-item).
    \item \textbf{Partial loop unrolling}: The 99-round VDF loop is annotated with \texttt{\#pragma unroll 3} ($99 = 33 \times 3$), reducing branch overhead while keeping register pressure manageable.
    \item \textbf{Work-group size hint}: \texttt{\_\_attribute\_\_((reqd\_work\_group\_size(256,1,1)))} enables the compiler to optimize register allocation and occupancy for a fixed work-group geometry.
    \item \textbf{Address-space correctness}: BLAKE3 initialization vectors are copied from \texttt{\_\_constant} to private (stack) memory before use, ensuring compatibility with strict OpenCL implementations (notably NVIDIA).
\end{itemize}

\subsection{Multi-GPU Support}

The miner supports multi-GPU configurations via per-device \texttt{GPUContext} instances. Each GPU maintains independent adaptive work sizing, cached challenge/target state, and persistent buffers. Nonce ranges are partitioned across devices proportionally to compute-unit count, and dispatches iterate sequentially with first-solution-wins semantics. Future work targets truly parallel per-GPU threads with channel-based solution aggregation.

\subsection{ASIC Resistance Analysis}

The BLAKE3+VDF design provides moderate ASIC resistance through two mechanisms:

\begin{enumerate}
    \item \textbf{Sequential VDF tail}: The 100-round sequential hash chain bounds parallelism. An ASIC can reduce per-hash latency but cannot parallelize the VDF chain within a single nonce.
    \item \textbf{Algorithmic agility}: The kernel source is compiled at runtime via OpenCL, enabling hash function upgrades (e.g., transitioning to a post-quantum hash) through height-gated consensus changes without hardware obsolescence.
\end{enumerate}
