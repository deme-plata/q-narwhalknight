# v10.9.55: Bitcoin Bridge Endpoints Disabled by Default
## Why all `/api/v1/bitcoin/*` routes return 503 unless `Q_BRIDGE_PRODUCTION_READY=1`
### Date: 2026-05-18 | Status: SHIPPED in v10.9.55

---

## Summary

Codex security audit 2026-05-18 (commit `c8cde7e41`) found **6 CRITICAL + 2 HIGH issues** in the Bitcoin bridge feature that prevent safe production use. v10.9.55 ships the consensus-critical BalanceRoot fixes + sync MVP, but the Bitcoin bridge endpoints are gated behind an env var (`Q_BRIDGE_PRODUCTION_READY`) that defaults to OFF. Until v10.9.56 closes these issues, real BTC must not be accepted.

## The 6 CRITICAL findings (from Codex audit)

1. **Captured Bitcoin RPC credentials in plaintext on the wire.** tcpdump on Epsilon 2026-05-18 captured `base64(qnk:QnkBtcBridge2026)` in cleartext. An on-path attacker (ISP, BGP hijacker, MITM) has wallet-control access. Mitigation requires WireGuard mesh + credential rotation.

2. **Concurrent finalize_lp_intent double-credit.** No per-intent lock or compare-and-set transition. Two simultaneous `POST /finalize` calls can both pass status checks and double-execute Steps 1-3 (refund QUG, mint wBTC, add liquidity). G1 fix in commit `c8cde7e41` closed only the crash-retry fall-through, not the concurrency race.

3. **Finalize crash windows non-idempotent.** Crashes between Step 1 (QUG refund) and Step 5 (Completed save) can leave partial state. Retries can re-refund or re-mint.

4. **mint_lp_first_or_proportional partial-rollback fund loss.** Helper debits QUG and wBTC inside, then validates pool. Failures between debit and credit leave the user with no LP tokens AND no escrow.

5. **create_lp_intent crash-recovery gap.** QUG debit, BTC address generation, and intent persistence are not atomic. Crash between debit and save = QUG lost with no recoverable intent.

6. **Atomic swaps non-functional.** `bank_btc_privkey: None` (TODO). Claim/refund methods synthesize fake txids, no real Bitcoin transaction construction. Swaps can be proposed but cannot safely complete BTC-side settlement.

Plus 2 HIGH:
- Cancel double-refund (save_intent failure ignored after balance refund).
- DepositBridge in-memory only — restart loses pending deposits, intents become unfinalizable.

## Why disable rather than fix in v10.9.55

- BalanceRoot enforcement at block 20,000,000 (~22 days from current tip) is the real ship deadline.
- Fixing all 6 CRITICAL findings is ~6-10 hours of careful work plus testing — not safely doable in the same release as the consensus changes.
- The bridge wallet is currently empty (`txcount=0` on Delta verified 2026-05-18 09:30 CEST). No real money at risk YET.
- The first-mover risk is that any future deposit lands in a wallet whose RPC creds are already leaked. Gating the endpoints prevents new deposits until cred rotation + WireGuard mesh close the leak window.

## What the gate does

In `crates/q-api-server/src/main.rs` near the bitcoin route registrations:

```rust
.merge(if std::env::var("Q_BRIDGE_PRODUCTION_READY").map(|v| v == "1").unwrap_or(false) {
    axum::Router::new()
        .route("/api/v1/bitcoin/swap", post(create_atomic_swap))
        // ... full bitcoin route block ...
} else {
    axum::Router::new()
        .route("/api/v1/bitcoin/bridge/status", get(|| async {
            Json(json!({ "success": false, "code": "BRIDGE_DISABLED_V10_9_55", "error": "..." }))
        }))
})
```

When `Q_BRIDGE_PRODUCTION_READY != "1"`:
- All 17 `/api/v1/bitcoin/*` routes (swap, deposit, withdraw, lp-intent family) return 404 (not registered)
- ONE status endpoint at `/api/v1/bitcoin/bridge/status` returns a structured 503-like JSON explaining the disable reason. The frontend can detect this and surface a clear "Bitcoin bridge temporarily unavailable" UI instead of bare 404s.

## When to re-enable

Set `Q_BRIDGE_PRODUCTION_READY=1` ONLY after ALL of:

1. **v10.9.56 WireGuard mesh deployed** across all 4 Quillon servers + Delta — see `docs/v10.9.56-wireguard-mesh-design.md`
2. **Bitcoin RPC credentials rotated** — new password in `/.env` on every node, bitcoin.conf rpcauth updated, old creds disabled
3. **Delta bitcoin.conf `rpcbind`** moved from `0.0.0.0` to `127.0.0.1,10.42.0.4` — public port 8332 dark
4. **Concurrency lock added** to `finalize_lp_intent` (CRITICAL #2)
5. **Idempotent state machine** for finalize Steps 1-3 (CRITICAL #3)
6. **`mint_lp_first_or_proportional`** wrapped in pre-validate + full-rollback (CRITICAL #4)
7. **`create_lp_intent`** persists pending intent before BTC address generation (CRITICAL #5)
8. **Atomic swap signing** implemented OR atomic swap endpoints stay disabled even when bridge is on (CRITICAL #6)
9. **Cancel idempotency** marker added (HIGH #1)
10. **DepositBridge persistence** at-create / on-status-change + startup load (HIGH #2)

Until ALL ten are done, `Q_BRIDGE_PRODUCTION_READY=1` should ONLY be set on test/canary environments, never on Beta/Gamma/Delta/Epsilon mainnet.

## Operator runbook for re-enabling on testnet

```bash
# On a NON-MAINNET test node:
echo 'Environment="Q_BRIDGE_PRODUCTION_READY=1"' \
  >> /etc/systemd/system/q-api-server.service.d/bridge-test.conf
systemctl daemon-reload
systemctl restart q-api-server

# Verify endpoints respond:
curl http://localhost:8080/api/v1/bitcoin/bridge/status
# Expected: structured response from get_bridge_status (not the disabled stub)

# To disable again:
rm /etc/systemd/system/q-api-server.service.d/bridge-test.conf
systemctl daemon-reload
systemctl restart q-api-server
```

## What stays accessible regardless

- `/api/v1/bitcoin/bridge/status` always responds — frontend uses this to detect disabled state
- All non-bitcoin endpoints (DEX, mining, balance, sync, transactions) unaffected
- The bridge code itself stays in the binary; only routes are gated. No conditional compilation, no feature flags.

## Reverting the gate (after v10.9.56 ships)

Remove the `.merge(if ... else ...)` wrapper around the bitcoin route block and restore the inline `.route()` chain. The bridge endpoints become unconditionally enabled. At that point, all 10 prerequisites above must be true.
