**Date:** 2026-05-15
**Test:** `/root/qnk-test-v10.9.27-beta/` (Beta), 38-minute run
**Outcome:** Partial success → stall at height 25,906
**Resolution:** v10.9.28 fixes (commits `ea5ccaaa` 10K cap, `87598f4e` metric wiring)

## Root cause of the stall

Two layered defaults that didn't agree:

1. **Scheduler dispatch cap** (`Q_GENESIS_LOOKAHEAD_BLOCKS`, default **1,000,000**)
   `turbo_sync.rs:3597` — keeps scheduled chunks within 1M of contiguous.

2. **Ingestion safety cap** (hardcoded **10,000**)
   `turbo_sync.rs` `[GAP SKIP REFUSED]` path — refuses any batch whose start height exceeds `contiguous_height + 10_000`.

The scheduler dispatched chunks across the full 1M window. About 51% of them came back at heights >10K beyond contiguous and got refused at ingest:

```
1,654 [GAP SKIP REFUSED] events in 38 minutes
1,738 successful Saved blocks events
Refusal rate: 1654 / (1654+1738) = 48.7%
```

When the test stalled, the contiguous tip was at 25,906 and the NEXT batch in the queue was 100,441-105,859 — a 74,534-block gap. Every subsequent batch landing at high heights got refused; chunks within 10K of 25,906 were no longer being dispatched (already drained from queue); progress halted.

## The v10.9.28 fix (committed `ea5ccaaa`)

Three sites in `turbo_sync.rs` (3597, 4228, 6948): `unwrap_or(1_000_000)` → `unwrap_or(10_000)`.

After the fix:
- Scheduler cap (10K) = Ingestion cap (10K)
- Every dispatched chunk lands within the ingestable window
- 0 refused batches expected
- Steady contiguous-tip advance
- Operators wanting parallelism still set `Q_GENESIS_LOOKAHEAD_BLOCKS=1000000` explicitly

## The v10.9.28 metric wiring fix (committed `87598f4e`)

The first slice of metric wiring: in the 5-second `[HEIGHT RECONCILE]` task at `main.rs:16740`, after fetching DB height:
```rust
if let Some(ref m) = app_state_height_reconcile.network_metrics {
    m.local_height.set(db_height as i64);
    if db_height > prev {
        m.blocks_synced_total.inc_by(db_height - prev);
    }
}
```

Once v10.9.28 lands, `qnk_local_height` will track real progress. Other gauges (`qnk_chunks_in_flight`, `qnk_chunk_retry_total`, `qnk_bootstrap_dial_total`, `qnk_block_pack_request_total`, `qnk_client_throttle_total`, `libp2p_swarm_*`) still need their hooks — tracked as v10.9.29 follow-up.

## What this test taught us

1. **The flow-control client semaphore + retry dampener (Steps 1+2) DID work** — the retry storm was much smaller than v10.9.26 because ClientThrottle didn't burn retry budget. The remaining 1,654 refusals were all from the layered-default bug, not client overrun.

2. **Connection stability is fine** — Epsilon and Gamma both connected and stayed for chunk delivery. The "flap" pattern earlier was WARMUP DRIVE noise. Real chunk deliveries are clean.

3. **The /metrics endpoint must lie loudly, not silently** — Operators (and ops sub-agents) trusted the gauges and concluded "sync is broken" when it was actually working at half speed. The wiring gap was the bigger UX failure than the ingest cap mismatch.

4. **The 38-min test produced ~25K blocks** — extrapolating naively at 40 blocks/sec average, full mainnet genesis sync (18M blocks) would take ~125 hours. After the v10.9.28 fix (eliminating the 49% wastage), that drops to ~63 hours. Still too slow for end-users — peak-burst suggests potential of ~5 hours if we can keep the scheduler saturating the ingest pipeline. That's v10.9.30+ work.

## Recommendations beyond v10.9.28

- **Wire all remaining metric gauges** — operators need an honest dashboard, not just `local_height`.
- **Investigate why throughput collapsed after the burst** — 990 blocks/sec then 40 blocks/sec average suggests the scheduler runs out of contiguous-adjacent chunks and starts pacing on chunk timeouts. The dispatch cadence needs to keep N chunks always in flight.
- **Validate the connection-flap diagnosis was wrong** — earlier analysis pointed at peer disconnects; deep log review shows the REAL issue was the layered cap mismatch. Past memory.md entries about "connection flap" should be revisited.
