# Investigation: Network-Wide libp2p / Gossipsub Mesh Failure
## v10.9.56 candidate — `peers=0`, `NoPeersSubscribedToTopic`, "no mesh peers" for every topic
### Date: 2026-05-18 | Status: ROOT CAUSE PROPOSED, awaiting live confirmation | Risk: ZERO (read-only diagnosis)

---

## 1. Executive summary

The network's libp2p mesh failed to form across all four operator-controlled nodes. The most direct on-disk evidence — Delta logging `NoPeersSubscribedToTopic` while `peers=0`, Epsilon's mesh-status warning firing for every topic with `peers=2`, Gamma at `peers=2`, Beta down — is consistent with **stale bootstrap peer IDs in `release/v10.9.55`'s `HARDCODED_BOOTSTRAP_PEERS` list combined with one HTTP-discovery endpoint pointing at the wrong port for Gamma**. The result is `WrongPeerId` rejections from the libp2p Noise handshake before any gossipsub subscription announce can be exchanged, so the mesh never forms even when TCP packets flow.

The fixes for both halves of this bug already exist on `agent/cross-shard-simd-validation` (commits `e80b0c63`, `2129fca1`) and in `crates/q-api-server/src/main.rs` on the v10.9.55 release branch. They were NOT carried into `crates/q-network/src/unified_network_manager.rs` on the release branch — that file is still pinned to peer IDs that were invalidated during the 2026-05-09 Epsilon restart, the 2026-05-16 Gamma zombie reset, and the 2026-05-17 Delta zombie restart.

This is a **fix-the-source-not-the-code** investigation. The codebase has all the right helpers already (HTTP discovery, `add_peer_address` learning from Identify, auto-reconnect with fresh URLs). They just got short-circuited by the peer-ID constants and a stale HTTP port.

The hardcoded `12D3KooWFpbXxxZJ…` Epsilon ID may also no longer match the live production node — `MEMORY.md`'s `epsilon_peer_id.md` says the real production ID is `12D3KooWSgic95Zm…` as of 2026-05-16. This needs live verification (see §5, recommended next steps) but if true would explain why Epsilon's mesh status is degraded even though Epsilon itself runs the v10.9.55 binary.

The topic strings are derived from `network_config.network_id` via `gossipsub_topic_prefix()` and the per-topic helpers in `crates/q-types/src/lib.rs:3948-3998`. For `NetworkId::MainnetGenesis`, the prefix is `/qnk/mainnet-genesis`. That hard-coded AI prefix is why Delta's `NoPeersSubscribedToTopic` warning fires for `qnk/ai/heartbeat/v1`: every node in mainnet-genesis subscribes to the same string, so the topic identity should match across peers. The bug is upstream — there ARE no peers.

### 3.3 Peer-ID validation handshake — `crates/q-network/src/handshake_validator.rs`

This is the application-level handshake that runs AFTER libp2p's transport-level Noise handshake completes. Three failure modes drop the connection (`unified_network_manager.rs:4401-4424`):

```
4401-4405: IncompatibleProtocol → close_connection
4407-4411: WrongNetwork         → close_connection
4413-4416: GenesisMismatch      → close_connection
4418-4422: MissingFeatures      → close_connection
```

`HandshakeMessage.network_id` is set from `validator.our_network_id` (`unified_network_manager.rs:2463`), which is `network_config.network_id.display_name().to_string()`. For mainnet-genesis that's the string `"Q-NarwhalKnight Mainnet Genesis"` (defined at `crates/q-types/src/lib.rs:3877`). All nodes on the same `NetworkId` enum value produce the same string here — this is consistent and not the bug.

`genesis_hash` is the hardcoded constant in `NetworkConfig::mainnet()` (`crates/q-types/src/lib.rs:4287-4294`) — an ASCII-prefixed 32-byte constant, identical across all binaries that link the same `q-types` version. Also consistent. Not the bug.

**Conclusion**: the application-level handshake is NOT the silent-failure path. Its failures emit loud `warn!` lines at 4402, 4408, 4414, 4419, 4432-4441. Those messages do NOT appear in the user-reported logs.

### 3.4 libp2p Noise transport handshake — silent path

If the transport-level Noise XX handshake fails because the peer-ID in our dialed multiaddr doesn't match the peer's actual identity-key derivation, the connection is rejected by libp2p BEFORE any application layer runs. The failure surfaces as:

```
unified_network_manager.rs:3494: libp2p::swarm::DialError::WrongPeerId { obtained, address }
```

which logs ERROR messages starting with `🚨 WRONG PEER ID MISMATCH:` (line 3495). These would be the canonical evidence; user has not (yet) reported observing them in the live logs but should grep for them.

This is the silent-failure path most consistent with `peers=0` while TCP works.

But the port is hardcoded `8080` at line 1527 AND line 1639. CLAUDE.md (verified 2026-05-11) and `crates/q-api-server/src/main.rs:504` agree that **Gamma's HTTP server is on port 8808, not 8080**. So:

- HTTP discovery to `89.149.241.126:8080` (Epsilon) — works.
- HTTP discovery to `5.79.79.158:8080` (Delta) — works.
- HTTP discovery to `185.182.185.227:8080` (Beta) — would work when Beta is up; currently times out.
- HTTP discovery to `109.205.176.60:8080` (Gamma) — **fails** (refused/timeout). UNM falls back to the stale hardcoded `12D3KooWFfZ…`. Outbound dial to Gamma's actual peer (`12D3KooWHNh…`) gets `DialError::WrongPeerId`. Connection drops.

This bug is symmetric: if Gamma is the dialer, it discovers others via HTTP and self-heals. If anyone dials Gamma, they fall back to the stale ID.

## 5. Hypothesis ranked list

Confidence column: HIGH = code path identified end-to-end, refute step trivial. MEDIUM = code path identified, may not be sole cause. LOW = consistent with symptoms, multiple pre-conditions.

### H1 — Stale hardcoded peer IDs in UNM cause `WrongPeerId` rejections. **Confidence: HIGH.**

Statement: `release/v10.9.55`'s `crates/q-network/src/unified_network_manager.rs:257,259` hardcodes Delta as `12D3KooWLJJ…` and Gamma as `12D3KooWFfZ…`, but their REAL current peer IDs are `12D3KooWPg1…` and `12D3KooWHNh…` respectively. Every libp2p dial to those nodes fails with `DialError::WrongPeerId`, and the connection never reaches gossipsub.

Confirm: on any node, grep journalctl for `WRONG PEER ID MISMATCH` (logged at ERROR level by `unified_network_manager.rs:3495`). Each such line says which peer was attempted with which expected ID and which was obtained. If line counts for Delta or Gamma peer-IDs are non-zero, confirmed.

Refute: if `WRONG PEER ID MISMATCH` count is zero across all nodes for the relevant peer IDs, H1 is refuted as a primary cause and we must look elsewhere.

### H2 — Gamma HTTP discovery uses port 8080 instead of 8808 → fallback to stale hardcoded ID. **Confidence: HIGH.**

Statement: `unified_network_manager.rs:274` ("`http://109.205.176.60:8080`") and `unified_network_manager.rs:1527` (hardcoded `8080`) both probe the wrong port for Gamma. CLAUDE.md and `q-api-server/src/main.rs:504` agree Gamma serves on 8808. Therefore HTTP-discovery for Gamma always fails, and UNM falls back to the stale hardcoded `12D3KooWFfZ…`, which itself fails Noise (H1). Compounds H1.

Confirm: grep journalctl for `HTTP error status` or `Client build error` from `fetch_peer_id_from_http` for `109.205.176.60`. Should see retries (`Retry attempt 2/3`) failing on Gamma when other nodes succeed.

Refute: if HTTP discovery to `109.205.176.60` is logged as successful (e.g., `Successfully fetched peer ID on attempt 1/3: 12D3KooWHNh…`), H2 is refuted — maybe Gamma has a port-8080 listener too. (Unlikely but possible.)

### H3 — Epsilon's REAL peer ID is `12D3KooWSgic95Zm…`, NOT the hardcoded `12D3KooWFpb…`. **Confidence: MEDIUM.**

Statement: per `MEMORY.md` → `epsilon_peer_id.md` (verified 2026-05-16 03:13 CEST via `https://quillon.xyz/api/v1/status`), Epsilon's production node identity is `12D3KooWSgic95Zm…`. All four files (UNM, main.rs, q-miner, GUI) hardcode `12D3KooWFpb…`. If the memory file is current, every peer dialing Epsilon's hardcoded multiaddr gets `WrongPeerId`.

The reason this is MEDIUM rather than HIGH: the memory file is a note, and Epsilon's identity may have changed AGAIN since 2026-05-16 (the file even acknowledges multiple instances on the host — `12D3KooWKa753…`, `12D3KooWFpb…`, `12D3KooWSgic…`). The live truth must be re-verified before committing to this hypothesis.

Confirm: `ssh root@89.149.241.126 "curl -s http://127.0.0.1:8080/api/v1/status | python3 -c \"import sys,json; print(json.load(sys.stdin)['data']['peer_id'])\""` and compare against `12D3KooWFpbXxxZJQ4FX9FGXrE5vaeNTCnZmLn6bqToRCMuiMpxM`.

Refute: if the live ID matches the hardcoded one, H3 is fully refuted (good news — one fewer thing to fix).

### H4 — Beta being offline removes the highest-eligibility DHT anchor. **Confidence: LOW (contributing).**

Statement: Beta (`12D3KooWKyj…`, line 261 — CORRECT) was the historical DHT/gossipsub anchor. With Beta down, the mesh has only three nodes to connect among. Combined with H1/H2 dropping two of those three, that leaves Delta ↔ Epsilon as the only edge, and a 1-edge graph at the gossipsub mesh layer can produce the observed `peers=2` on each side (one is libp2p-level outgoing + one inbound from the same peer, depending on counters).

Confirm: bring Beta back up; observe whether mesh recovers without code change. Refute: if Beta stays down but mesh recovers after H1+H2 patches, H4 is contributory but not blocking.

### H5 — Gossipsub `ValidationMode::Strict` + `flood_publish` interaction starves new peers. **Confidence: LOW.**

Statement: `unified_network_manager.rs:1897` sets `ValidationMode::Strict`, which rejects unsigned messages. New peers joining the mesh might be unable to flood-publish until their signing keys are accepted via Subscribe events. This is gossipsub-spec normal behavior, but tunable mesh parameters (`mesh_n_low=4` minimum, currently from `(mesh_n / 2).max(4)`) might be too aggressive for a 3-node network.

Confirm: temporarily set Q_GOSSIPSUB_MESH_N=2 and Q_GOSSIPSUB_FLOOD=true; observe if mesh forms.

Refute: irrelevant if H1/H2 are confirmed — strict validation doesn't prevent Noise from completing.

### H6 — Run-loop starvation of `swarm.next()` by `block_pack_response_rx`. **Confidence: LOW (refuted by symptoms).**

Statement: the `biased;` order in `run_once()`'s `tokio::select!` (line 5523-5670) prioritizes block-pack responses over swarm events. Under heavy sync activity, swarm events could be starved.

Refute: at `peers=0`, there are no block-pack requests, so the channel is empty. `command_rx` for routine commands does run, but each `run_once()` call services exactly one branch and returns. The outer loop in `main.rs:6244-6297` re-enters at most every yield, so swarm.next() is reached frequently in this regime. Inconsistent with `peers=0`.

## 9. Closing notes

This is the second time the mesh has fallen over for peer-id-staleness reasons in the same month (Gamma 2026-05-16, now this). The pattern is identical: a node restarts with a fresh DB → libp2p generates a new identity → all other nodes' hardcoded multiaddrs are stale → silent `WrongPeerId` rejection → mesh degrades silently. Recommendation C.1 (load bootstrap from a JSON file refreshed via the live `/api/v1/status` endpoint) is not optional from a reliability standpoint; it's load-bearing.

The codebase already has the right ingredients (HTTP discovery, Identify learning, auto-reconnect). They were defeated by:
1. ONE port number (`:8080` instead of `:8808` for Gamma).
2. TWO outdated peer IDs in the const list.
3. The release branch having been cut from the SIMD branch BEFORE the fold-in of those exact fixes.

All three are mechanical to fix. The lesson is process: a pre-release diff against the live agent branch's UNM should be mandatory.
