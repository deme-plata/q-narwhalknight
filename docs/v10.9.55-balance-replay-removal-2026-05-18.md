# v10.9.55: Removal of Convergence Migration v103
## Why we deleted a chain-replay balance migration on a sparse chain
### Date: 2026-05-18 | Status: SHIPPED in v10.9.55

---

## Summary

`crates/q-storage/src/lib.rs::safe_batched_convergence_v103` was a migration designed to delete all `wallet_balance_*` keys from RocksDB and rebuild them by replaying `qblock:height:1..=tip`. It was triggered automatically at startup on `chain_rebuild_enabled && !checkpoint_applied` nodes, and operators were instructed by a divergence-check log message to "delete the migration flag and restart" whenever balance hashes drifted from peers.

**It has been removed.** This document records why, because in 2028 we will want to remember.

---

## Why the migration was dangerous

### 1. The chain is empirically sparse

Direct read-only `ldb scan` on Epsilon's `cf_blocks` on 2026-05-18 produced:

- **8,173,472 `qblock:height:` keys** present
- **Range 201 → 18,130,600**, current tip
- **~9.96M missing heights** (55% of the range)

The full breakdown is in `docs/technical-review-sparse-chain-truth-v1.md`. The post-15M range is 93-96% dense (DAG-Knight protocol-natural sparsity); pre-7M is ~3% dense (historical damage from March 2026 compaction loss + April 2026 v10.2.8 cleanup bug).

### 2. The replay silently skips missing heights

`safe_batched_convergence_v103` iterates `for height in 1..=tip`, reads `qblock:height:N`, and on missing keys does `Ok(None) => continue`. This means on Epsilon the replay processes only ~45% of the height range and rebuilds balances missing the other 55%.

### 3. Those missing heights contained real mining rewards

Epsilon's authoritative wallet balances were earned by approximately 100,000 Contabo VPS instances mining at ~8 GH/s collective compute for ~3 months. That work paid for actual electricity. Those balances are not abstract numbers — they are PoW-backed claims with audit trail in the chain itself.

A replay-based rebuild on the sparse chain would have destroyed real PoW-backed value by silently dropping all rewards from missing heights.

### 4. The misleading log message

When peer balance hashes diverged from the local hash (a periodic check at `state_sync_api.rs:~1054`), the error message read:

> `Run convergence migration to fix: delete RocksDB flag 'migration_safe_convergence_v103_done' and restart`

This was honest at the time it was written — the migration was assumed safe because the chain was assumed contiguous. But by 2026-05-18 we know empirically that the chain is sparse and the advice is dangerous on Epsilon. An operator following the message would have permanently destroyed authoritative state.

### 5. CLAUDE.md rules already prohibit this

`CLAUDE.md` "BALANCE INTEGRITY — NON-NEGOTIABLE RULES" Rules 1 and 3:

> **Rule 1**: `save_wallet_balances` MUST be max-wins. Any code that writes wallet balances to RocksDB MUST check `existing >= new` first and skip the write if the existing value is higher. `save_wallet_balances` is a batch overwrite — it will silently destroy correct higher balances if called with stale/partial data.

> **Rule 3**: Epsilon's wallet balances are authoritative — never overwrite them.

The migration violated both. The fact that it was authored before these rules existed does not make it safer.

## What was intentionally KEPT

| Location | Why |
|---|---|
| `has_migration_flag` / `delete_migration_flag` generic helpers | Used by other migrations; the bug was in v103-specific code, not the helpers |
| `state_sync_api.rs:980-988` QUGUSD ghost-prevention gate | Now permanently in conservative mode (flag never set ⇒ QUGUSD updates from peers rejected post-bootstrap) — safer behavior |
| `state_sync_api.rs:~1597` second QUGUSD gate | Same reasoning |

The QUGUSD gate effectively becomes "always conservative." This is a tightening, not a loosening — and matches CLAUDE.md "feedback_no_balance_replay" memory: clean-resync, not in-place repair.

## Operational recovery procedure when balance hashes diverge

The OLD advice: "delete the convergence migration flag and restart."
The NEW advice: don't replay. Audit. Clean-resync.

### If you see the divergence-check error in journalctl:

1. **DO NOT** delete any RocksDB flag.
2. **DO NOT** run any migration that touches wallet_balance_* keys.
3. **DO**: determine which side is correct.
   - Pick a wallet present on both sides (e.g., the founder wallet).
   - Audit its balance via independent block-replay arithmetic from a known checkpoint.
   - The side that matches the audit is correct; the other side has drifted.
4. If THIS node is wrong:
   - Stop the q-api-server cleanly (SIGTERM + 120s timeout — see `v10.9.55-systemd-hardening.md`).
   - Move the current `data-mainnet-genesis` directory aside (don't delete).
   - Create an empty DB.
   - Restart and sync from a known-good peer (Beta or Epsilon).
   - Verify the founder wallet balance matches after sync.
5. If PEER is wrong:
   - Continue serving normally.
   - Inform the operator of the divergent peer; they should follow step 4 on their end.

## Future-proofing

To make this class of bug harder to re-introduce in v10.9.56+:

1. **Single helper** for bulk wallet-balance deletion that REFUSES on `is_genesis_node()` and requires explicit operator-confirmation env var on other nodes. Replace all 10 call sites with this helper.
2. **Compile-time lint** (a custom clippy lint or a build-script grep) that fails the build if `delete_by_prefix(b"wallet_balance_")` is called outside the single guarded helper.
3. **Regression test** as above.
