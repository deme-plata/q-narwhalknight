# v10.9.56 Design: WireGuard Mesh between Quillon Nodes
## Encrypt inter-server traffic — Bitcoin RPC creds, gossipsub control, deploy commands
### Date: 2026-05-18 | Status: DESIGN (implementation deferred from v10.9.55 — too risky live)

---

## 1. Problem

Verified empirically 2026-05-18 via tcpdump on Epsilon during a live `getblockchaininfo` RPC call to Delta:8332:

> ⚠️ base64(`qnk:QnkBtcBridge2026`) found in plaintext in the pcap. The cred is trivially decodable.

The Bitcoin RPC link between Quillon nodes (Beta, Gamma, Epsilon) and the centralized bitcoind on Delta uses HTTP Basic Auth in cleartext. Anyone on the path — ISP, datacenter, peering point, BGP hijacker, MITM — can passively capture the credentials.

Same threat applies to:
- Inter-Quillon HTTP state sync (`/api/v1/sync/full-state` on port 8080) — no TLS by default
- gossipsub control messages (libp2p has its own encryption — Noise protocol — but the IP-level traffic is observable)
- `ha-deploy.sh` SCP operations between Beta and the other servers

## 2. Goal

Wrap all inter-Quillon-server traffic in WireGuard, terminating each endpoint at a single 10.x.x.x address per server. No application code change required — the underlay just gets encrypted.

## 3. Topology

Full mesh, 4 servers:

```
            Beta (185.182.185.227)
                10.42.0.1
              /     |     \
             /      |      \
   Gamma  --       |       -- Delta
   (109.205.176.60) |      (5.79.79.158)
   10.42.0.2       |       10.42.0.4
             \     |     /
              \    |    /
            Epsilon (89.149.241.126)
                10.42.0.3
```

Each server has a WireGuard interface `wg0` with:
- One private key (per server, generated on the server, never leaves)
- Peer entries for the 3 other servers (their public keys + endpoint IPs + allowed-IPs 10.42.0.x/32)
- AllowedIPs scoped tightly (per-peer single /32) so traffic for 10.42.0.x is routed only via wg0

## 4. Address allocation

| Server | Public IP | wg0 IP | Notes |
|---|---|---|---|
| Beta | 185.182.185.227 | 10.42.0.1 | Production primary |
| Gamma | 109.205.176.60 | 10.42.0.2 | HA backup |
| Epsilon | 89.149.241.126 | 10.42.0.3 | 10Gbit supernode |
| Delta | 5.79.79.158 | 10.42.0.4 | Mainnet2026.1.1 + bitcoind |

10.42.0.0/24 reserved. Room for 250+ future Quillon nodes.

## 5. Per-server config

Identical pattern, just changes which peer entries are included.

**Beta `/etc/wireguard/wg0.conf`** (root only, mode 600):

```ini
[Interface]
PrivateKey = <BETA_PRIVATE_KEY>
Address = 10.42.0.1/24
ListenPort = 51820
SaveConfig = false
PostUp = iptables -A INPUT -i wg0 -j ACCEPT
PostDown = iptables -D INPUT -i wg0 -j ACCEPT

# Gamma
[Peer]
PublicKey = <GAMMA_PUBLIC_KEY>
Endpoint = 109.205.176.60:51820
AllowedIPs = 10.42.0.2/32
PersistentKeepalive = 25

# Epsilon
[Peer]
PublicKey = <EPSILON_PUBLIC_KEY>
Endpoint = 89.149.241.126:51820
AllowedIPs = 10.42.0.3/32
PersistentKeepalive = 25

# Delta
[Peer]
PublicKey = <DELTA_PUBLIC_KEY>
Endpoint = 5.79.79.158:51820
AllowedIPs = 10.42.0.4/32
PersistentKeepalive = 25
```

Repeat the pattern on Gamma/Epsilon/Delta with their own private key and the other 3 peers' public keys.

## 6. Application config changes (post-mesh)

After WireGuard is up and 10.42.0.x pings work between all four servers, change application configs to use the mesh addresses:

### 6.1 Bitcoin RPC

| File | Current | After |
|---|---|---|
| `/.env` on Beta/Gamma/Epsilon | `BTC_RPC_URL=http://5.79.79.158:8332` | `BTC_RPC_URL=http://10.42.0.4:8332` |
| Delta `bitcoin.conf` `rpcbind` | `0.0.0.0` | `127.0.0.1,10.42.0.4` (only localhost + wg) |
| Delta `bitcoin.conf` `rpcallowip` | public IPs of Beta/Gamma/Epsilon | `127.0.0.1, 10.42.0.0/24` |

Result: bitcoind no longer listens on the public 5.79.79.158:8332. Any attacker probing port 8332 gets refused. The RPC creds, even if leaked, are useless from outside the WireGuard mesh.

### 6.2 Inter-Quillon HTTP state sync

| File | Current | After |
|---|---|---|
| Bootstrap peer entries in `main.rs:~530` + `CLAUDE.md` | public IPs | `10.42.0.x` |
| state-sync fetch URLs | `http://185.182.185.227:8080/...` | `http://10.42.0.1:8080/...` |

### 6.3 ha-deploy.sh

| File | Current | After |
|---|---|---|
| SCP destinations | `root@5.79.79.158`, etc. | `root@10.42.0.x` |
| SSH host keys | per public IP | per wg IP (need to re-trust on first use) |

## 7. Firewall changes

Per server, add a rule for the WireGuard UDP port:

```bash
# Allow inbound WG from the 3 known peer IPs only (not the world)
iptables -A INPUT -p udp --dport 51820 -s 185.182.185.227 -j ACCEPT  # Beta
iptables -A INPUT -p udp --dport 51820 -s 109.205.176.60 -j ACCEPT   # Gamma
iptables -A INPUT -p udp --dport 51820 -s 89.149.241.126 -j ACCEPT   # Epsilon
iptables -A INPUT -p udp --dport 51820 -s 5.79.79.158 -j ACCEPT      # Delta
iptables -A INPUT -p udp --dport 51820 -j DROP                       # default deny
```

Persist via `iptables-save` or wherever the server's firewall config lives.

## 8. Migration sequence (zero-downtime)

```
T-0:    Generate WG keypairs on each server. Distribute public keys via secure
        channel (Beta has ops control; SSH the pubkeys, never private keys).
T+1hr:  Install wireguard on each server (apt install wireguard wireguard-tools).
T+1hr:  Apply /etc/wireguard/wg0.conf on each server.
T+1hr:  Bring up wg0 on each: systemctl enable --now wg-quick@wg0
T+1hr:  Verify 10.42.0.x ping between all pairs.
        At this point WG is up but NOTHING uses it yet. Public IPs still work.
T+2hr:  On Delta only: add 10.42.0.4 to bitcoin.conf rpcbind (keep 0.0.0.0 too
        for now — both addresses listen). Reload bitcoind via SIGHUP.
T+2hr:  Verify from Beta: curl through wg works → bitcoind responds.
T+3hr:  Update /.env BTC_RPC_URL on Beta/Gamma/Epsilon to 10.42.0.4:8332.
        Restart q-api-server one node at a time, verify bridge calls.
T+4hr:  Soak test 1 hour. Verify capture: tcpdump on the public Delta:8332 port
        should show NO traffic from Quillon nodes anymore.
T+5hr:  On Delta only: change bitcoin.conf rpcbind from "0.0.0.0,10.42.0.4" to
        just "127.0.0.1,10.42.0.4". Restart bitcoind. Now port 8332 is dark to
        the public internet.
T+6hr:  Update Quillon main.rs bootstrap peers to use 10.42.0.x. Ship as v10.9.56.
T+7hr:  Update ha-deploy.sh SCP destinations to 10.42.0.x.
T+8hr:  Done. All inter-Quillon traffic on WireGuard.
```

Public-internet exposed surface drops from 4 ports per node (8080, 9001, 8332, 51820) to 3 (8080, 9001, 51820 — and 51820 is UDP with strict peer allowlist).

## 9. Rollback plan

If something breaks at any stage:

| Stage | Rollback |
|---|---|
| 1-2 | `systemctl stop wg-quick@wg0` — no apps are using it yet, safe |
| 3 | Revert /.env to public BTC_RPC_URL, restart q-api-server. Bitcoin bridge resumes on public path |
| 4 | Same |
| 5 | Reload bitcoind with rpcbind=0.0.0.0 again. Public access restored |
| 6+ | Revert Quillon binary, re-deploy v10.9.55 |

WireGuard does NOT modify any state on the servers — it's purely a routing/encryption layer. Removing it is a `systemctl stop` away.

## 10. Why this is v10.9.56 not v10.9.55

- 4 servers × ~1hr per server = 4hr of careful coordinated config
- Needs operator attention through migration sequence steps
- A single misconfigured peer entry disconnects nodes from each other
- v10.9.55 already covers BalanceRoot pre-20M-activation (real deadline)
- WireGuard is operational hardening — no consensus dependency

## 11. Open questions for v10.9.56 implementation

1. Should we use `wg-quick@wg0` (the systemd unit) or boot wireguard via systemd-networkd? The former is simpler, the latter integrates better with systemd's lifecycle.
2. Should we add a 5th address (10.42.0.5) for Alpha (canary Docker host) so canary releases can join the mesh? Currently Alpha is on a different model (no public IP for the docker container).
3. Persistent-keepalive interval — 25 seconds is the default. Lower → faster NAT hole-punching, more bandwidth. Higher → quieter, NAT might time out the punch. 25s is fine for the four-server case.
4. Should bitcoin's ZMQ ports (28332-28334) also be wg-restricted? Currently exposed to 0.0.0.0. ZMQ has no auth — anyone can subscribe to raw block/tx feeds. Privacy concern, not a security one (data is public anyway). Recommend yes, restrict to wg.
