### Option 2: Multi-Validator Setup (Advanced)

**Perfect for:**
- Production networks
- Multiple server deployments
- High-availability setups
- Distributed consensus

#### Required Environment Variables

| Variable | Description | Example |
|----------|-------------|---------|
| `Q_VALIDATOR_INDEX` | Unique validator ID (0, 1, 2, ...) | `0`, `1`, `2` |
| `Q_TOTAL_VALIDATORS` | Total validators in network | `3`, `5`, `10` |
| `Q_DB_PATH` | Unique database path per validator | `./data-validator-0` |
| `--node-id` | Unique node identifier | `validator-0` |

#### Example: 3-Validator Network

**Validator 0 (Primary - Produces empty blocks)**:
```bash
Q_VALIDATOR_INDEX=0 \
Q_TOTAL_VALIDATORS=3 \
Q_DB_PATH=./data-validator-0 \
./q-api-server-v0.0.22-beta --port 8080 --node-id validator-0
```

**Validator 1 (Secondary)**:
```bash
Q_VALIDATOR_INDEX=1 \
Q_TOTAL_VALIDATORS=3 \
Q_DB_PATH=./data-validator-1 \
./q-api-server-v0.0.22-beta --port 8081 --node-id validator-1
```

**Validator 2 (Tertiary)**:
```bash
Q_VALIDATOR_INDEX=2 \
Q_TOTAL_VALIDATORS=3 \
Q_DB_PATH=./data-validator-2 \
./q-api-server-v0.0.22-beta --port 8082 --node-id validator-2
```

#### Critical Requirements ⚠️

1. **Only Validator 0 produces empty blocks** (prevents DAG width explosion)
2. **All validators accept mining solutions** and produce blocks with transactions
3. **Each validator MUST have**:
   - Unique database path (`Q_DB_PATH`)
   - Unique port number
   - Unique node-id
   - Sequential validator index (0, 1, 2, 3... no gaps)

### Advanced Configuration Options

#### Block Production Control

**Q_BLOCK_INTERVAL_SECS**
- Default: `15` seconds
- Range: `5-300` seconds
- Example: `Q_BLOCK_INTERVAL_SECS=30`

**Q_MIN_SOLUTIONS_PER_BLOCK**
- Default: `1`
- Minimum mining solutions before block production
- Example: `Q_MIN_SOLUTIONS_PER_BLOCK=10`

**Q_MAX_SOLUTIONS_PER_BLOCK**
- Default: `100`
- Maximum: `1000`
- Example: `Q_MAX_SOLUTIONS_PER_BLOCK=500`

**Q_ALLOW_MANUAL_TRIGGER**
- Default: `false` (security)
- Enable manual block triggering API endpoint
- Example: `Q_ALLOW_MANUAL_TRIGGER=true`

## 🚀 Why This Matters

### Problem Solved: Competing Empty Blocks

**Without proper validator configuration**:
```
Time T=15s, all 3 validators produce block at height 10:

Validator 0: [Empty Block 10-A]
Validator 1: [Empty Block 10-B] ← WASTE
Validator 2: [Empty Block 10-C] ← WASTE

Result: 3x storage waste, 3x bandwidth waste, DAG width explosion
```

**With validator coordination** (v0.0.22-beta):
```
Time T=15s:

Validator 0: [Empty Block 10-A] ✅ (designated producer)
Validator 1: (waits for transactions)
Validator 2: (waits for transactions)

Result: Only ONE empty block, efficient DAG continuity
```

## 📊 Configuration Validation

The v0.0.22-beta binary includes **automatic configuration validation**:

### Startup Checks ✅

1. **Block interval**: Must be 5-300 seconds
2. **Solution limits**: `min ≤ max ≤ 1000`
3. **Port configuration**: Valid port range
4. **Validator index**: Must be `< total_validators`
5. **Multi-validator warning**: If `total_validators > 1`
6. **Security warning**: If `allow_manual_trigger = true`

### Validation Example
```bash
# This will FAIL with error message:
Q_BLOCK_INTERVAL_SECS=3 ./q-api-server-v0.0.22-beta --port 8080

Error: Invalid block interval: 3 seconds (must be 5-300)

# This will SUCCEED:
Q_BLOCK_INTERVAL_SECS=15 ./q-api-server-v0.0.22-beta --port 8080
```

## 🔍 Technical Context

### Why Environment Variables for Validators?

**Design Decision**: Use environment variables instead of config files

**Advantages**:
1. ✅ **Container-friendly**: Docker, Kubernetes, systemd
2. ✅ **Cloud-native**: AWS, GCP, Azure environment variable support
3. ✅ **No file management**: No need to distribute config files
4. ✅ **Override priority**: Environment variables override defaults
5. ✅ **Security**: Secrets management integration (HashiCorp Vault, AWS Secrets Manager)

**Example: Docker Deployment**
```dockerfile
# Validator 0
docker run -e Q_VALIDATOR_INDEX=0 \
  -e Q_TOTAL_VALIDATORS=3 \
  -e Q_DB_PATH=/data \
  -v validator0-data:/data \
  -p 8080:8080 \
  q-narwhalknight:v0.0.22-beta

# Validator 1
docker run -e Q_VALIDATOR_INDEX=1 \
  -e Q_TOTAL_VALIDATORS=3 \
  -e Q_DB_PATH=/data \
  -v validator1-data:/data \
  -p 8081:8080 \
  q-narwhalknight:v0.0.22-beta
```
