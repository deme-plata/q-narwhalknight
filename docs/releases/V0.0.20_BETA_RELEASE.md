# Q-NarwhalKnight v0.0.20-beta Release Notes

## Release Date: October 25, 2024

## Critical Bug Fix: Peer Count Display

### Overview
This release fixes a critical display bug where the frontend Explorer page showed "0 active peers" even when libp2p was successfully discovering and connecting to multiple peers.

### What Was Fixed

#### Root Cause
The `ConnectionEstablished` event handler in `unified_network_manager.rs` was updating the `discovered_peers` HashSet but **NOT** updating the `connected_peer_count` atomic counter that the API endpoint reads from.

**Code Flow (Before Fix):**
1. libp2p discovers peers via mDNS and Kademlia DHT ✅
2. `ConnectionEstablished` event fires and adds peer to `discovered_peers` set ✅
3. Log shows "📊 [NETWORK STATE] Total discovered peers: 3" ✅
4. **BUG**: Atomic counter NOT updated ❌
5. API `/api/v1/status` reads from atomic counter → always returns 0 ❌
6. Frontend displays 0 peers ❌

#### Fix Applied

**File**: `crates/q-network/src/unified_network_manager.rs`

**Lines 445-451** - Added atomic counter update in ConnectionEstablished:
```rust
let mut peers = self.discovered_peers.write().await;
peers.insert(peer_id);
let peer_count = peers.len();
drop(peers); // Release lock before atomic operations

// Update atomic counter for API endpoint (thread-safe)
self.connected_peer_count.store(peer_count, std::sync::atomic::Ordering::SeqCst);
```

**Lines 459-469** - Added atomic counter update in ConnectionClosed:
```rust
SwarmEvent::ConnectionClosed { peer_id, .. } => {
    // Remove peer from discovered set
    let mut peers = self.discovered_peers.write().await;
    peers.remove(&peer_id);
    let peer_count = peers.len();
    drop(peers);

    // Update atomic counter
    self.connected_peer_count.store(peer_count, std::sync::atomic::Ordering::SeqCst);

    info!("👋 [DISCONNECTION] Connection closed with peer: {} (remaining peers: {})", peer_id, peer_count);
}
```

### Verification

After restart, logs now show:
```
📊 [NETWORK STATE] Total discovered peers: 3 (atomic counter updated)
```

And API returns the correct count:
```bash
curl http://localhost:8080/api/v1/status | jq '.data.connected_peers'
# Returns actual peer count (e.g., 3)
```

### Technical Notes

- **Atomic Ordering**: Uses `Ordering::SeqCst` for sequentially consistent atomic operations
- **Lock Management**: Properly releases write lock with `drop()` before atomic operations to avoid deadlocks
- **Thread Safety**: Atomic counter is lock-free and safe to read from multiple threads
- **Event Handling**: Both connection established and closed events update the counter
