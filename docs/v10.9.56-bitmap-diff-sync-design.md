# v10.9.56 Design: Bitmap-Diff Sync for Sparse Chains
## Replacing the range-fetch model with set-based block requests
### Status: DESIGN — to be implemented after v10.9.55 ships and is Codex-green-lit

---

## 1. Executive summary

v10.9.55 ships a working but inefficient sync model for the sparse chain:
1. Requester asks peer for blocks in a range `[N, N+W)`
2. Peer returns whatever exists in that range (sparse subset)
3. Requester advances `synced_through` to N+W and loops

This works — a fresh node terminates eventually — but it's bandwidth- and time-inefficient on the actual chain (8.17M present heights spread across [201, 18.1M], average density ~45%):
- Re-fetches blocks the requester already has on partial syncs
- Cannot tell peer "skip these"
- Pays a 12-second timeout cost when a window happens to land entirely on missing heights
- Cannot parallelize across peers safely (overlap creates duplicate work)

v10.9.56 replaces this with a **bitmap-diff** protocol:
1. Peers exchange compressed `present_heights` bitmaps over a small range window
2. Requester computes `wanted = peer.bitmap AND NOT my.bitmap AND target_window`
3. Requester sends explicit `GetBlocksByHeights({h1, h2, h3, ...})` requests
4. Peer returns exactly those blocks

Idempotent, no duplicate fetches, parallelizable, no 12-second misses. Expected sync-time improvement: **100x+ on the early sparse range, 5-10x on the dense recent range**.

## 2. Why now, why this design

### 2.1 Why now
- v10.9.55 shipped the per-height marker layer (`h_present:NNN` in CF_MANIFEST) — the data layer this design depends on
- Mainnet at 18.13M, BalanceRootV1 activates at 20M (~22 days), but sync correctness is a chronic operational pain point — fresh nodes consistently fail to bootstrap from genesis. v10.9.56 is the operational sync release; consensus fixes shipped in v10.9.55.
- 100M-block projection in ~1 year. The range model's cost scales with the number of dead heights (linear-ish); the bitmap model's cost scales with the number of WANTED blocks (sub-linear after the one-time bitmap exchange).

### 2.2 Why roaring bitmap

Considered alternatives:
- **Plain `Vec<u64>` of heights** — ~64 MB for 8M heights, no compression. Killed by bandwidth.
- **Bloom filter** — fast existence check but probabilistic; false positives turn into 12-second wasted requests. Killed by correctness.
- **Run-length encoded ranges** — works for dense chunks but the chain has both extremes (post-15M dense + pre-7M heavy holes). RLE blows up on the irregular sparse regions.
- **Roaring bitmap (`roaring` crate or `croaring-rs`)** — purpose-built for "billions of u32s with mixed density." Uses bitmap, array, or run-length containers per 16-bit chunk adaptively. Empirically 1-2 MB compressed for our 8M present heights.

Roaring also has the right semantics: AND, OR, NOT, AND-NOT are all O(small) per chunk. The `wanted = peer AND NOT mine` operation is fast.

Pinning: `roaring = "0.10"` (stable, maintained, MIT). For u64 heights specifically we use `roaring::Treemap` (which is a sorted map of `u32` → `RoaringBitmap` to extend the address space to 64-bit).

## 3. Wire protocol

### 3.1 New request variants

Extend `crates/q-types/src/block_pack.rs` enums. The existing `BlockPackRequest` becomes one variant of a wider enum, or two new variants are added alongside it:

```rust
// crates/q-types/src/block_pack.rs

#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub enum SyncRequest {
    /// Pre-v10.9.56 range request — fallback when peer doesn't support bitmap.
    BlockPackRange {
        start_height: u64,
        limit: usize,
    },
    /// v10.9.56: get the peer's present-heights bitmap for a window.
    /// Window MUST be at most BITMAP_WINDOW_MAX (1,000,000 heights) to bound
    /// response size and avoid OOM on either side.
    GetBitmapSummary {
        range_start: u64,
        range_end: u64,  // exclusive
    },
    /// v10.9.56: get specific blocks by explicit height list.
    /// MUST be ≤ MAX_HEIGHTS_PER_REQUEST (1,000) to bound response size.
    GetBlocksByHeights {
        heights: Vec<u64>,  // sorted ascending recommended; server doesn't require
    },
}

#[derive(Debug, Clone, PartialEq, Eq, Serialize, Deserialize)]
pub enum SyncResponse {
    /// Existing range response (carries the v10.9.41+ permanent_gap field).
    BlockPackRange(BlockPackResponse),
    /// v10.9.56: serialized roaring bitmap of present heights in the requested
    /// window. Empty bytes ⇒ peer has zero blocks in that window.
    BitmapSummary {
        range_start: u64,
        range_end: u64,
        roaring_bytes: Vec<u8>,  // RoaringTreemap::serialize_into output
    },
    /// v10.9.56: sparse response to GetBlocksByHeights. Length ≤ request length;
    /// peer may return fewer if some heights were unavailable (race or pruning).
    BlocksByHeights {
        blocks: Vec<QBlock>,
    },
}
```

### 3.2 Limits

| Constant | Value | Rationale |
|---|---|---|
| `BITMAP_WINDOW_MAX` | 1,000,000 heights | Bounds bitmap response to ~200 KB even at extreme density |
| `MAX_HEIGHTS_PER_REQUEST` | 1,000 | Average block size ~30 KB → ~30 MB max response, fits in single libp2p message |
| `BITMAP_FRESHNESS_TTL` | 60 seconds | Bitmap re-fetched after this; balances tip-following vs request count |

### 3.3 Bitmap serialization

```rust
use roaring::RoaringTreemap;

fn serialize_bitmap(bm: &RoaringTreemap) -> Vec<u8> {
    let mut buf = Vec::with_capacity(bm.serialized_size());
    bm.serialize_into(&mut buf).expect("infallible into Vec");
    buf
}

fn deserialize_bitmap(bytes: &[u8]) -> Result<RoaringTreemap> {
    RoaringTreemap::deserialize_from(bytes)
        .context("invalid bitmap bytes")
}
```

The `roaring` crate's serialization format is stable across versions (documented as the "portable" format). Cross-version compatibility is preserved if we pin the crate version.

## 4. Component design

### 4.1 Storage layer additions

`crates/q-storage/src/lib.rs`:

```rust
/// Serialize the present-heights bitmap for a window.
///
/// Reads `h_present:NNN` markers in CF_MANIFEST and falls back to scanning
/// `qblock:height:N` for pre-v10.9.55 blocks (where markers don't exist yet).
/// O(window_size) one-time per call; caller caches the result.
pub async fn build_present_heights_bitmap(
    &self,
    range_start: u64,
    range_end: u64,
) -> Result<RoaringTreemap>;

/// Multi-get blocks by explicit height list. Returns sparse Vec.
pub async fn get_qblocks_by_heights(
    &self,
    heights: &[u64],
) -> Result<Vec<QBlock>>;

/// Backfill markers for blocks stored before v10.9.55. Idempotent;
/// triggered at startup if marker count < contiguous_height. Writes
/// markers in batches of 10,000 with progress logging.
pub async fn backfill_present_height_markers(&self) -> Result<u64>;  // returns count written
```

### 4.2 Sync layer (`crates/q-storage/src/turbo_sync.rs`)

New entry point `sync_to_height_bitmap` invoked when peer supports v10.9.56:

```rust
pub async fn sync_to_height_bitmap(&self, target_height: u64) -> Result<()> {
    let start = self.storage.get_synced_through_height();
    let mut local_bitmap = self.storage.build_present_heights_bitmap(0, target_height).await?;

    let mut window_start = start;
    while window_start < target_height {
        let window_end = (window_start + BITMAP_WINDOW_MAX).min(target_height);

        // 1. Pick a peer (round-robin or beta-trust-weighted)
        let peer = self.pick_peer_for_bitmap_sync().await?;

        // 2. Fetch peer's bitmap for this window
        let peer_bitmap = self.request_bitmap(peer, window_start, window_end).await?;

        // 3. Compute diff
        let mut wanted: RoaringTreemap = peer_bitmap.clone();
        wanted -= &local_bitmap;  // peer AND NOT mine

        // 4. Batch the wanted heights into MAX_HEIGHTS_PER_REQUEST chunks
        let wanted_vec: Vec<u64> = wanted.iter().collect();
        for chunk in wanted_vec.chunks(MAX_HEIGHTS_PER_REQUEST) {
            let blocks = self.request_blocks_by_heights(peer, chunk).await?;
            self.ingest_blocks(blocks).await?;
            // Update local bitmap incrementally so we don't re-request mid-window
            for b in &blocks {
                local_bitmap.insert(b.header.height);
            }
        }

        // 5. Advance synced_through to window_end (range completed)
        self.storage.advance_synced_through(window_end).await?;
        window_start = window_end;
    }
    Ok(())
}
```

### 4.3 Network layer (`crates/q-network/src/unified_network_manager.rs`)

Add handlers for the new request variants:

```rust
SyncRequest::GetBitmapSummary { range_start, range_end } => {
    if range_end - range_start > BITMAP_WINDOW_MAX {
        return SyncResponse::error("window too large");
    }
    let bm = storage.build_present_heights_bitmap(range_start, range_end).await?;
    SyncResponse::BitmapSummary {
        range_start,
        range_end,
        roaring_bytes: serialize_bitmap(&bm),
    }
}
SyncRequest::GetBlocksByHeights { heights } => {
    if heights.len() > MAX_HEIGHTS_PER_REQUEST {
        return SyncResponse::error("too many heights");
    }
    let blocks = storage.get_qblocks_by_heights(&heights).await?;
    SyncResponse::BlocksByHeights { blocks }
}
```

### 4.4 Protocol negotiation

Two options. Recommend **B**.

**A. libp2p protocol version string** — add `/qnk/sync/v2` alongside existing `/qnk/sync/v1`. Peers advertise both via the libp2p identify protocol; requester picks v2 when available. Cleanest but requires touching the libp2p behaviour wiring.

**B. Probe-and-fallback** — requester sends `GetBitmapSummary { start: 0, end: 1 }` as a capability probe. v10.9.56 peers respond with valid bitmap bytes (even if empty). Pre-v10.9.56 peers respond with deserialize error or an error variant; requester falls back to range mode for that peer. Per-peer fallback state is cached in the peer registry (`supports_bitmap: bool`) so probes happen once per peer per session.

Option B is simpler — no protocol-string changes, works with the existing request-response wiring.

## 5. Backfill strategy

8.17M existing blocks have no `h_present:` markers. The marker-only fast path in v10.9.55's `is_height_present` falls back to `qblock:height:N` lookup, so correctness isn't affected, but `build_present_heights_bitmap` for a range that overlaps the pre-v10.9.55 era pays the fallback cost: 8M lookups at ~1ms each = ~2 hours per bitmap build.

Mitigation: one-time backfill at startup if `marker_count < contiguous_height`. Steps:

1. At storage open, count markers via `scan_prefix("h_present:")` (~30 sec on Epsilon).
2. If count < `contiguous_height - 10_000` (with some slack), trigger backfill in a background tokio task.
3. Backfill scans `qblock:height:` keys in 10K batches; for each found height, write the marker via a small WriteBatch.
4. Estimated time on Epsilon: 8M markers × ~50µs per write batched at 10K/batch = ~7 minutes.
5. Log progress every 1M markers; expose count via `/metrics`.
6. `build_present_heights_bitmap` runs the fallback path during backfill — slower but correct. After backfill completes, it's the fast marker path only.

This is a one-time cost per node. Acceptable.

## 7. Test plan

### 7.1 Unit tests
- `bitmap_roundtrip` — serialize, deserialize, assert equality with random sparse bitmaps
- `bitmap_diff_correctness` — populate two bitmaps, compute AND-NOT, verify expected set
- `backfill_idempotent` — run backfill twice on the same DB; second run is a no-op (no marker rewrites)
- `is_height_present_marker_vs_fallback` — verify both code paths return identical answers

### 7.2 Integration tests
- `bitmap_sync_full_range` — two storage engines, populate one with sparse pattern, sync the other via bitmap-diff; assert byte-identical blocks
- `bitmap_sync_against_pre_v10956_peer` — peer rejects bitmap request; requester falls back to range mode; sync still succeeds
- `bitmap_sync_parallel_peers` — three peers; split the diff across them; verify no duplicates ingested

### 7.3 Production canary
- Deploy v10.9.56 to Alpha Docker only
- Fresh-sync test: empty DB → sync from Epsilon
- Compare wall-clock time and bytes-transferred against v10.9.55 baseline
- Target: 10x faster sync end-to-end, <50% of v10.9.55 bandwidth

### 7.4 Adversarial
- Malicious peer sends a bitmap claiming heights it doesn't have. Requester asks for them via `GetBlocksByHeights`; peer returns empty. Requester decrements peer trust and routes future requests elsewhere.
- Malicious peer sends massive bitmap (>BITMAP_WINDOW_MAX serialized). Server-side cap rejects before transmission; client-side cap rejects on receive.

## 8. Risks and open questions

### 8.1 Memory pressure during backfill
8M marker writes batched at 10K per batch is fine (~80 batches), but if we go larger (e.g., 100K per batch) we hit RocksDB write buffer limits. Pin at 10K/batch with explicit `Flush` between batches.

### 8.2 Bitmap freshness at tip
Tip-following nodes advance their bitmap continuously. If a requester fetched a peer's bitmap 60 seconds ago, the peer has new blocks the requester won't know about. Two solutions:
- (a) `BITMAP_FRESHNESS_TTL = 60s` — refetch after that
- (b) Combine bitmap-diff for historical sync with gossipsub-based tip-following (already in place)

Recommend (b) — bitmap-diff is for catching up; gossipsub handles live tail.

### 8.3 Bitmap exchange triggers per-peer probe latency
First probe to a peer adds ~50-200ms before any blocks come back. Mitigation: probe all known peers in parallel at startup, cache `supports_bitmap` in the registry, so subsequent sync_to_height_bitmap invocations have zero probe overhead.

### 8.4 What about reorgs?
If the requester accepts a block at sparse height N and later a reorg removes it, the marker must also be removed. Add: when block deletion is performed (reorg only — NEVER from sync paths per the v10.2.8 lesson), also delete the marker in the same WriteBatch. Existing `chain_reorganization.rs` is the call site.

### 8.5 Open question: should bitmap exchange be gossipsub instead of request-response?
Gossipsub broadcasts a peer's bitmap to all neighbors periodically. Pros: requester always has fresh-ish peer bitmaps without dedicated probes. Cons: bandwidth amplification (every peer sees every other peer's bitmap), and `N` peers × `M` neighbors × ~1MB bitmap = expensive at scale.

Recommend: stick with request-response for the initial release. Re-evaluate when peer count grows.

### 8.6 Open question: backfill in the background or block startup?
Background is more user-friendly (node starts serving immediately) but `build_present_heights_bitmap` is slow until backfill completes. If a peer asks for our bitmap during the first 5 minutes of uptime, we either:
- Refuse with "still indexing" — clean but peers retry and waste cycles
- Serve a partial bitmap from whatever markers exist — incorrect (peer thinks we don't have blocks we actually have, requests them again, peer-trust score for us suffers)
- Serve the bitmap built from the fallback path (slower but correct) — first request is slow but correct

Recommend: third option. Slow first bitmap exchange (~30s for full chain on Epsilon) is acceptable for the rare case of "fresh restart + bitmap request in same minute."

## 10. What this DOESN'T solve

- **Pre-7M historical loss** — those blocks are gone from every node, not unreachable. Bitmap-diff just means we don't waste time asking for them; it doesn't recover them. Sparse-chain-truth doc Section 9 stands.
- **DAG-Knight protocol-natural sparsity** — the chain WILL have ~5% naturally-missing heights post-15M forever. Bitmap-diff is what makes that fine.
- **Consensus correctness** — that's the v10.9.55 BalanceRoot fixes. v10.9.56 is operational only.
- **Block validation chain semantics** — if validation requires prev_block_at(N-1).hash chaining instead of DAG-hash lookup, bitmap-diff still wedges. Codex's PASS 2 review of v10.9.55 includes checking this assumption (question C); v10.9.56 inherits the same dependency.

## 11. Decision points the operator must make before implementation

| Decision | Recommendation |
|---|---|
| `roaring = "0.10"` vs `croaring-rs` (C bindings) | `roaring` (pure Rust, no FFI, slightly slower but simpler) |
| Protocol negotiation: probe-and-fallback vs new `/qnk/sync/v2` | Probe-and-fallback (B) |
| Backfill at startup: block or background | Background, with first bitmap-request paying fallback cost if early |
| `BITMAP_WINDOW_MAX = 1_000_000` | OK; revisit if bandwidth becomes an issue |
| `MAX_HEIGHTS_PER_REQUEST = 1_000` | OK |
| Bitmap-via-gossipsub | Defer; stick with request-response |
