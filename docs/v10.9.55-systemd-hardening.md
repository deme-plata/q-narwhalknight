# v10.9.55 systemd hardening — operator runbook

**Status:** Per-server config change. Apply during the v10.9.55 deploy window or as a separate maintenance step. No code change; no binary needed.

## What this prevents

The March 2026 RocksDB compaction-loss incident on Epsilon (which destroyed ~8M historical blocks in the pre-7M range) was caused by `kill -9` interrupting in-flight SST writes during compaction. Systemd's default `KillMode=control-group` plus a short `TimeoutStopSec` can produce the same outcome whenever a node restart pile-up or a slow shutdown occurs.

The companion `paranoid_file_checks` and `force_consistency_checks` flags added in v10.9.55's `crates/q-storage/src/kv.rs` provide *detection* of corruption at startup. The systemd changes here provide *prevention* by giving RocksDB enough time to flush WAL + finish any in-progress compaction output before the OS process is hard-killed.

## Required diff

Add or change three directives in the `[Service]` section:

```diff
 [Service]
 Type=simple
 WorkingDirectory=/
 ExecStart=/q-api-server-v10.9.55 --port 62003
 EnvironmentFile=/.env
 Restart=on-failure
 RestartSec=5
 LimitNOFILE=65536
 KillSignal=SIGTERM
-TimeoutStopSec=30
+KillMode=mixed
+TimeoutStopSec=120
+SendSIGKILL=yes
```

What each does:

- **`KillMode=mixed`**: Sends SIGTERM to the main process ONLY, leaving child processes alone. With the default `control-group`, all processes in the cgroup get SIGTERM simultaneously, which can interrupt forked compaction helpers mid-write. Mixed lets the main process complete the graceful shutdown sequence first, then systemd cleans up any remaining children.
- **`TimeoutStopSec=120`**: Up from 30s. On Epsilon with a 232 GB DB and 48,306 SST files, a clean shutdown can take 30-60s just to flush the WAL and quiesce compaction. 120s leaves head-room without locking the operator out of restart commands for too long.
- **`SendSIGKILL=yes`**: Keeps the default forced-kill behavior explicit after the 120s grace period. systemd logs a stop-timeout warning when this fires; operators must monitor `journalctl -u q-api-server` for those messages to know the graceful shutdown was overridden by SIGKILL. Setting it to `no` would prevent the SIGKILL entirely, which can wedge the unit forever in `stopping` if the process refuses to exit — leave it at the default (`yes`) and rely on the journal alert.

## Apply procedure (one server at a time)

For each of Beta, Gamma, Delta, Epsilon:

```bash
# 1. Edit the service file
vi /etc/systemd/system/q-api-server.service

# 2. Reload systemd to pick up the change (does NOT restart the node)
systemctl daemon-reload

# 3. Verify the changes are loaded
systemctl show q-api-server | grep -E "KillMode|TimeoutStopUSec|SendSIGKILL"
# Should print:
#   KillMode=mixed
#   TimeoutStopUSec=2min
#   SendSIGKILL=yes

# 4. (Optional) Verify graceful shutdown works BEFORE relying on it for v10.9.55 deploy.
#    On a non-production server, run:
systemctl stop q-api-server
# In another terminal: journalctl -u q-api-server -f
# You should see RocksDB compaction-finish events before the process exits.
```

## What NOT to do

- **Do NOT use `kill -9` to stop the node** — that's what got us here. Use `systemctl stop q-api-server` (which sends SIGTERM and respects TimeoutStopSec).
- **Do NOT reduce TimeoutStopSec below 60s** on Epsilon — the 232 GB DB legitimately needs the time. On smaller-DB servers (Delta), 60s is fine.
- **Do NOT restart all four servers at once** — apply this change one server at a time, verify shutdown is clean, then proceed to the next. The HA rolling-deploy via `ha-deploy.sh` is the canonical pattern.

## Verification after deploy

After all four servers have the new systemd config + v10.9.55 binary:

```bash
# Each server, in turn:
journalctl -u q-api-server --since "1 hour ago" | grep -E "compaction_finished|paranoid"
# Should see normal compaction events. NO "ERROR: file checksum mismatch" or
# "Corruption: missing SST file" entries. If you see corruption, force-consistency-checks
# will refuse to start the node — that's the new safety net working.
```

If a node fails to start after v10.9.55 with `force_consistency_checks=true` rejecting the DB, the corruption is real and pre-existing (not caused by v10.9.55). Restore from backup (`/home/orobit/data-mainnet-genesis/backups/` or via the storage_api restore endpoint) and re-sync from a healthy peer.
