# v10.9.56 Design Addendum: Concurrency, DoS, and Tip-Catch-Up Hardening
### Date: 2026-05-18 | Status: DESIGN DELTA on `v10.9.56-bitmap-diff-sync-design.md`

This addendum captures Codex's hardening suggestions on the 2026-05-18 design review of the v10.9.56 bitmap-diff sync proposal. The architecture is unchanged; these are implementation contracts that must be honored when the code is written.

---

## 1. RocksDB snapshot during bitmap build (concurrency hazard)

**Hazard**: While the one-time marker backfill is running in the background (~7 minutes on Epsilon), a peer can request our `BitmapSummary` for a window that overlaps the in-progress range. Without protection, the read sees a partial snapshot: some markers exist, some don't (because backfill is still inserting them). The response would falsely report missing heights, and the peer would either:
- Refuse to sync from us (degraded peer score), or
- Re-request those blocks needlessly.

**Contract**: `build_present_heights_bitmap(range_start, range_end)` MUST use a RocksDB read-snapshot taken at the start of the call. All marker lookups AND the fallback `qblock:height:N` lookups (for pre-v10.9.55 blocks) read from that snapshot. RocksDB snapshots are O(1) to take and have zero overhead until they're held for a long time.

**Implementation:**
```rust
pub async fn build_present_heights_bitmap(
    &self,
    range_start: u64,
    range_end: u64,
) -> Result<RoaringTreemap> {
    // Take a snapshot for consistent read across markers + fallback
    let snapshot = self.hot_db.snapshot();

    // Fast path: scan h_present: markers within range from the snapshot
    let mut bm = self.scan_markers_in_range(&snapshot, range_start, range_end).await?;

    // Fallback path: for any height in range that has no marker but does have a
    // qblock:height:N key, add it to the bitmap.
    // (Optimized away when backfill is complete — see metric below.)
    if !self.is_backfill_complete().await {
        self.augment_with_fallback(&snapshot, range_start, range_end, &mut bm).await?;
    }

    Ok(bm)
}
```

**Optimization**: If `is_backfill_complete()` returns true (all pre-v10.9.55 markers backfilled), skip the snapshot — marker-only read is sufficient and faster. The flag is set by the backfill task on its final batch and persisted under `cf_manifest::h_present_backfill_complete = b"1"`.

## 2. Bitmap deserialization size cap (DoS prevention)

**Hazard**: A malicious peer sends a crafted `RoaringTreemap` byte stream that decodes to a huge in-memory structure. The `roaring` crate's `deserialize_from` does not have a built-in cap; a few hundred KB of input can expand into hundreds of MB of memory if the container header claims many runs.

**Contract**: Before passing bytes to `RoaringTreemap::deserialize_from`, the client MUST check the byte length against a hard cap and the deserialized cardinality against a separate cap.

```rust
const MAX_BITMAP_BYTES: usize = 2 * 1024 * 1024;  // 2 MB on the wire
const MAX_BITMAP_CARDINALITY: u64 = 5_000_000;     // 5M present heights per window

fn deserialize_bitmap_safe(bytes: &[u8]) -> Result<RoaringTreemap> {
    if bytes.len() > MAX_BITMAP_BYTES {
        return Err(anyhow!(
            "peer bitmap exceeds wire-size cap: {} > {} bytes",
            bytes.len(), MAX_BITMAP_BYTES
        ));
    }
    let bm = RoaringTreemap::deserialize_from(bytes)
        .context("invalid bitmap bytes")?;
    if bm.len() > MAX_BITMAP_CARDINALITY {
        return Err(anyhow!(
            "peer bitmap claims {} present heights, exceeds cap {}",
            bm.len(), MAX_BITMAP_CARDINALITY
        ));
    }
    Ok(bm)
}
```

If either cap trips, the peer's bitmap response is rejected, the peer's trust score is decremented, and the requester falls back to range mode for that peer.

## 3. Required new metrics

In addition to the metrics already listed in section 6 of the parent design doc, add:

| Metric | Type | Description |
|---|---|---|
| `qnk_bitmap_backfill_in_progress` | gauge (0/1) | 1 while the marker backfill task is running. Operators can correlate slow bitmap requests with active backfill. |
| `qnk_sync_mode{peer="...",mode="range\|bitmap"}` | gauge | Per-peer current sync mode. Lets operators verify that fallback is happening as expected for pre-v10.9.56 peers. |
| `qnk_bitmap_window_fetch_seconds` | histogram | Time to fetch + deserialize a peer's bitmap for one window. Buckets: 100ms / 500ms / 1s / 5s / 30s. |
| `qnk_bitmap_rejected_total{reason="size\|cardinality\|cardinality_zero\|invalid"}` | counter | Count of peer bitmaps rejected by the size cap or other validation. Spikes suggest malicious peers. |
| `qnk_bitmap_diff_blocks_wanted` | histogram | Number of blocks in a `wanted` set after diff. Buckets: 0 / 10 / 100 / 1K / 10K / 100K. |

## 4. Final-window re-fetch for tip catch-up

**Hazard**: The current design fetches a peer's bitmap once per window. For the FINAL window (the one containing `target_height`), the peer may receive new blocks via gossipsub between the bitmap fetch and the last `GetBlocksByHeights` call. The requester would miss them and require another `sync_to_height_bitmap` invocation to pick them up.

**Contract**: After the `wanted` set for the final window is drained (i.e., all initially-requested blocks have been fetched), if `target_height` was the upper bound of the window, perform ONE more bitmap fetch on the same window and re-compute the diff. If the new diff is non-empty, fetch those blocks too. Repeat at most once per `sync_to_height_bitmap` invocation to avoid infinite loops in a busy network.

```rust
// Inside sync_to_height_bitmap, after the wanted set is drained:
if window_end >= target_height {
    let fresh_bitmap = self.request_bitmap(peer, window_start, window_end).await?;
    let mut tip_wanted: RoaringTreemap = fresh_bitmap.clone();
    tip_wanted -= &local_bitmap;
    if !tip_wanted.is_empty() {
        let tip_blocks: Vec<u64> = tip_wanted.iter().collect();
        let blocks = self.request_blocks_by_heights(peer, &tip_blocks).await?;
        self.ingest_blocks(blocks).await?;
    }
}
```

This adds at most one additional bitmap round-trip per full sync invocation. Worth it to avoid a separate 12-second range fallback or a second invocation cycle.

## 5. Backfill cursor (optional optimization)

**Note**: This is an OPTIONAL optimization to speed up backfill restart after a crash. Not required for correctness — the backfill is already idempotent because writing existing markers is a no-op.

If implemented: persist `h_present_backfill_cursor = NNNNNNNNNNN` under `CF_MANIFEST` each time a 10K-batch completes. On restart, the backfill task resumes from `cursor + 1` instead of restarting from height 0.

Trade-off: saves ~5-7 minutes on crash-restart at the cost of ~5 lines of code. Defer to implementation time; decision can be made then.

## 6. Probe-cache TTL (defer)

**Note**: Codex flagged that the per-peer `supports_bitmap: bool` cache is permanent within a session. If a peer hot-upgrades from v10.9.55 to v10.9.56 mid-session, our cache stays at `false` and we keep using range mode against them.

**Decision**: Acceptable for v10.9.56 launch. Operator restart of either node refreshes the cache. If hot-upgrade-without-restart becomes a routine operation, add a 10-minute re-probe interval in a follow-up release.

## 7. Summary of contracts that must hold during implementation

The implementer of v10.9.56 must satisfy ALL of:

1. ✅ `build_present_heights_bitmap` reads from a single RocksDB snapshot
2. ✅ `is_backfill_complete()` flag persisted in CF_MANIFEST; used to skip fallback augmentation
3. ✅ Bitmap deserialization checks `bytes.len() <= MAX_BITMAP_BYTES` AND `bm.len() <= MAX_BITMAP_CARDINALITY`
4. ✅ Rejected bitmaps decrement peer trust + fall back to range mode for that peer
5. ✅ Required new metrics (section 3) emitted via `/metrics`
6. ✅ Final-window re-fetch performed at most once per `sync_to_height_bitmap` invocation
7. ⚪ Backfill cursor — optional but encouraged
8. ⚪ Probe-cache TTL — deferred to a follow-up release
